# Sync hermes-superpowers to superpowers v6.0.3 — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring every skill in `hermes-superpowers` to content parity with upstream `superpowers` v6.0.3, while preserving the Hermes-native transformation layer that makes the skills work in the Hermes harness.

**Architecture:** This is a *content port*, not a code change. For each skill we take upstream v6.0.3 content as the source of truth, apply the Hermes Transformation Contract (rename, tool-mapping, cross-ref/path rewriting — defined in Global Constraints), and write the result into the Hermes layout (`superpowers/superpowers-<skill>/`). Hermes-specific artifacts (the `superpowers-slash-commands` dispatcher, README, LICENSE, .gitignore) are preserved/refreshed. Two upstream subsystems are deliberately **not** ported per scope decisions: the brainstorm visual-companion Node server (rewritten text-only) and the per-harness tool-mapping `references/` docs (rewritten Hermes-native).

**Tech Stack:** Markdown skills, a few bash helper scripts (SDD only), frontmatter YAML. Verification is grep/structural assertion + `git diff` against upstream, not unit tests.

## Global Constraints

The **Hermes Transformation Contract** below applies to EVERY ported file. Each task's requirements implicitly include all of it.

### T1 — Frontmatter `name:` prefix
`name: <skill>` → `name: superpowers-<skill>` (one skill = one `superpowers-<skill>` value; never the bare upstream name).

### T2 — Hermes-native tool mapping
Wherever the ported content names a tool or describes a file/IO action, use Hermes-native tool names instead of Claude/other-harness tools. Map:
- create/write a file → `write_file(path="...", content="...")`
- read a file → `read_file(path="...")`
- targeted/surgical edit → `patch(...)` (NOT `write_file` for whole existing files)
- search file contents → `search_files(...)`
- list/find files → `find_files(...)` / `ls`
- Claude `Skill` tool → `skill_view(name="superpowers-<skill>")`
- `TodoWrite` → Hermes task-list tool
- `EnterPlanMode` → Hermes plan mechanism (or remove if no Hermes equivalent)

**Density rule:** match the tool-naming style of the *current Hermes* version of that skill. Most Hermes skills explicitly insert tool calls where upstream uses generic prose ("Write the spec to…"). Keep upstream's new v6 *content/structure*, but apply these tool names wherever a tool or IO action is named.

### T3 — Skill cross-reference rewriting
- `superpowers:<skill>` → `superpowers-<skill>`
- `/superpowers:<skill>` → `/superpowers-<skill>`
- In prose, a skill the agent invokes is `superpowers-<skill>`; a bare `<skill>` token that is clearly a skill name should also gain the `superpowers-` prefix when it is an invocation/cross-ref.

### T4 — Path rewriting
- `skills/<skill>/<file>` → `superpowers/superpowers-<skill>/<file>`
- sibling refs `../<skill>/<file>` → `../superpowers-<skill>/<file>`
- `references/<platform>-tools.md` links → **removed/rewritten** (those files are not ported; see T6).

### T5 — Hermes-specific artifacts preserved
`superpowers/superpowers-slash-commands/` (Hermes-only dispatcher), root `README.md`, `LICENSE`, `.gitignore` are Hermes-specific — refresh, do not overwrite with upstream.

### T6 — Deliberately NOT ported (scope decisions)
- **Brainstorm visual-companion server**: `scripts/{server.cjs,helper.js,frame-template.html,start-server.sh,stop-server.sh}` are dropped. `visual-companion.md` is rewritten **text-only** (terminal-rendered visuals: mermaid, ASCII, tables — no Node server, no `--open`, no `start-server.sh`).
- **using-superpowers `references/` tool-mapping docs** (`claude-code-tools.md`, `codex-tools.md`, `copilot-tools.md`, `gemini-tools.md`, `pi-tools.md`, `antigravity-tools.md`): not ported. Tool-access/platform sections are rewritten Hermes-native.
- **All other-harness plugin manifests, hooks, tests, release-notes** (`.claude-plugin/`, `.codex-plugin/`, `hooks/`, `tests/`, `RELEASE-NOTES.md`, etc.): out of scope — Hermes ships only skills.
- **EXCEPTION — SDD scripts**: `subagent-driven-development/scripts/{task-brief,review-package,sdd-workspace}` ARE ported (skill-internal bash helpers ≤1.4 KB, required by the v6 SDD SKILL.md to function).

### T7 — Per-task verification gate (the "test")
Each task ends with a verification gate using these grep assertions (run from `~/other/hermes-superpowers/`):
```bash
# G1: frontmatter name is prefixed
grep -m1 "^name:" superpowers/superpowers-<skill>/SKILL.md   # → name: superpowers-<skill>
# G2: no leaked upstream namespace refs inside the skill
! grep -rn "superpowers:" superpowers/superpowers-<skill>/   # → exit 1 (no matches)
# G3: no broken references/ or brainstorm scripts/ links
! grep -rn "references/.*-tools.md\|start-server.sh\|stop-server.sh\|server.cjs" superpowers/superpowers-<skill>/
```
Then commit. Use `git diff 917e5f5 v6.0.3 -- skills/<skill>/` (in the upstream repo) as the content-parity reference while porting.
